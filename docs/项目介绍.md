Gstack 项目深度分析
一、项目概述
Gstack 是一个为 AI 编码助手（Claude Code、OpenAI Codex、Factory）提供能力的单仓框架，核心包括：

持久化无头浏览器 (Browse CLI)
结构化工作流技能库 (31 个技能模板)
代码生成管道 (模板 → 文档)
完整的评估基础设施
二、架构亮点
1. 技能即提示词模板
这是项目最核心的创新——整个技能系统用 Markdown 提示词模板实现，而非传统代码：


SKILL.md.tmpl (人工编写的模板)
       ↓
gen-skill-docs.ts (模板引擎)
       ↓
SKILL.md (自动生成，Claude 直接执行)
核心理念：SKILL.md 文件就是程序，由 Claude 阅读并执行。无需运行时、无需 VM、无需调度器。

2. 单一真相源文档生成
源文件	生成内容	验证方式
browse/src/commands.ts	命令参考表	构建时自动检查
browse/src/snapshot.ts	快照标志文档	同步验证
SKILL.md.tmpl	SKILL.md	--dry-run + git diff
这从根本上解决了文档与代码不同步的问题。

3. 差异化测试选择
每个 E2E 测试声明其依赖的文件模式：


// test/helpers/touchfiles.ts
export const E2E_TOUCHFILES: Record<string, TouchfilePattern[]> = {
  'ship-skill': ['ship/SKILL.md', 'test/skill-e2e-ship.test.ts'],
  'browse-qa': ['browse/src/**/*.ts', 'qa/SKILL.md'],
  // ...
}
运行时：

git diff 获取变更文件
匹配每个测试的 touchfile 模式
只运行相关测试
效果：30 分钟、$4 的评估套件 → 大多数 PR 只需 5 分钟

4. 多模型交叉审查管道
/autoplan 技能实现的三阶段审查：


阶段1: CEO 审查 (战略)
  ├─ Claude 子代理
  └─ OpenAI Codex
       ↓
阶段2: 设计审查 (UI/UX)
  ├─ Claude 子代理
  └─ OpenAI Codex
       ↓
阶段3: 工程审查 (架构)
  ├─ Claude 子代理
  └─ OpenAI Codex
决策分类：

Mechanical — 自动决定
Taste — 提交给人工审批
User Challenge — 永不自动决定
三、核心技术组件
Browse CLI 架构

┌─────────────────┐
│   CLI Binary    │ ~58MB 编译后单文件
├─────────────────┤
│   HTTP Server   │ Bun.serve() + Bearer 认证
├─────────────────┤
│   Playwright    │ → Chromium (无头)
└─────────────────┘
关键设计：

随机端口 (10000-60000) 避免 10 个并发工作区的端口冲突
版本自动重启（SHA 不匹配时杀死旧进程）
纯 HTTP 请求/响应（无 WebSocket，更低延迟）
Ref 系统 (元素定位)
通过可访问性树而非 DOM 变更：


// 1. 获取 ARIA 树
const tree = await page.accessibility.snapshot();

// 2. 分配 refs: @e1, @e2, @e3...

// 3. 后续操作
await click('@e3'); // 解析为 getByRole(role, { name }).nth(index)
优势：

✅ 无 CSP 违规
✅ 无框架水合冲突
✅ 支持 Shadow DOM
模板解析器系统
28 个解析器模块，分为：

模块	职责
preamble.ts	启动引导块（更新检查、会话追踪）
browse.ts	命令表、快照标志
design.ts	设计方法论、黑名单
composition.ts	技能组合
confidence.ts	置信度评分
learnings.ts	跨技能制度记忆
四、开发者体验创新
1. 实时监控模式

bun run dev:skill  # 监视 .tmpl 文件，自动重新生成
bun run skill:check  # 健康仪表板
bun run eval:compare  # 对比两次评估运行
2. Git Worktree 隔离测试

// lib/worktree.ts
WorktreeManager → 创建隔离的工作树
  → 复制构建产物
  → 测试代理在隔离环境中运行
  → 收集变更为补丁
  → SHA-256 去重
3. 原子文件操作

fs.writeFileSync(tmpPath, data);
fs.renameSync(tmpPath, finalPath);  // 防止崩溃后部分读取
4. 非致命可观测性
所有可观测性 I/O 都包装在 try/catch 中：

写入失败绝不会导致测试失败。测试本身才是真相来源；可观测性是尽力而为的。

五、对我们有何意义
1. 提示工程即系统编程
Gstack 证明了结构化提示词可以作为完整的应用程序。这为我们提供了一个新的编程范式：

技能是人类可读、可审计的
任何人都可以通过编辑 Markdown 来创建技能
"编译器"是简单的模板引擎
2. 文档驱动开发
单一真相源模式可用于任何项目：

代码是源，文档是派生
构建时自动验证一致性
杜绝文档与代码不同步
3. 差异化测试策略
对于大型项目，E2E 测试成本高昂。Gstack 的 touchfile 系统展示了如何：

智能选择测试子集
将 CI 成本降低 80%+
保持完整的安全覆盖
4. AI 协作模式
多模型审查管道展示了如何：

让 AI 模型互相审查
利用模型分歧作为人工判断的信号
建立完整的审计追踪
5. 崩溃恢复哲学
不尝试自愈，直接退出并重启

这比尝试连接到半死进程的浏览器更简单、更可靠。

七、可借鉴的模式
模式	适用场景
